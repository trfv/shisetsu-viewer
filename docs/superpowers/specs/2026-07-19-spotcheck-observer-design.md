# spot check オブザーバの正式化と区分フィルタ型サイトへの対応 設計

- 日付: 2026-07-19
- 対象: `packages/scraper/tools/spotcheck/observe.ts`（新設）、`.claude/commands/spot-check.md`（改修）
- 背景: 2026-07-19 の `/spot-check` 実行（Issue #1626）で、フェーズ 2 の観測手順に二つの欠陥が判明した

## 目的

`/spot-check` のフェーズ 2（サイト観測）を、再現可能で正しい手順に置き換える。

2026-07-18 の設計[^original]はフェーズ 2 を Playwright MCP でのライブ探索として定めた。
初回の実行で、この手順には二つの問題があった。

一つはコストである。
江戸川区の予約サイトを開いて a11y スナップショットを取ると、利用規約の全文が木構造として返る。
8 自治体を巡回するには、この規模のスナップショットを何十回も受け取ることになる。

もう一つは観測の誤りであり、こちらが重い。
豊島区と江戸川区と文京区の予約システムは、施設別空き状況の画面で「その他の条件で絞り込む」から時間帯を選び、区分ごとに表を描き直す。
スクレイパーはこのフィルタを区分の数だけ切り替えて読んでいる（`engines/webrGrand.ts` および `tokyo-edogawa/index.ts` の `extract`）。
ところが初回の実行では、フィルタを操作せずに描画される集約表をそのまま読んだ。
つまりスクレイパーが見ている画面とは別の画面を観測しており、豊島区が MATCH と判定されたのは、その日の記号がたまたま全区分で同じだったことによる。

判定器は正しく動いていた。
誤っていたのは判定器への入力である。
silent failure を検出する道具が、自身の入力の誤りに気付けない状態にあった。

[^original]: `docs/superpowers/specs/2026-07-18-agentic-spotcheck-design.md`。盲検（エージェントは D1 の期待値を読まない）と判定器の独立性（`symbolMap.ts` は scraper の `STATUS_MAP` を import しない）を中核に据えた 3 段パイプライン。

## スコープ

やること:

- 観測スクリプト `tools/spotcheck/observe.ts` の新設
- 区分フィルタ型サイトの読み取り戦略の実装
- 盲検の線引きをユニットテストで検査する仕組みの追加
- `spot-check.md` のフェーズ 2 の書き換え

やらないこと:

- `judge.ts`、`judgeReport.ts`、`symbolMap.ts` の変更。判定側の契約は不変とする。荒川区の記号 `O` の追加は凡例による裏取りが済んでいないため Issue #1626 に残す
- 全自治体での動作検証。汎用に書くが、検証は plan が引いたサンプルの範囲にとどめる
- observe.ts の定期実行。実サイトに接続するため、自己修復と同じく手動起動レーンに置く
- 観測の完全自動化。サイトのお知らせや注記の読解はエージェントに残す（後述）

## アーキテクチャ

3 段パイプラインの構成は変えない。
変わるのはフェーズ 2 の内部だけである。

```
plan.ts     → plan.json          決定論。変更なし
observe.ts  → raw/*.json         新設。prepare を借りて到達し、独立に読む
            → observed/*.json    cells / legend / url / screenshotPath を自動生成
            → screenshots/*.png
エージェント → observed/*.json    raw を読んで note を補い、異常を記録する
judge.ts    → SPOTCHECK_RESULT   決定論。変更なし
```

`observe.ts` は既存の `ObservedSample` 型（`judgeReport.ts`）に適合する JSON を書き出す。
判定側の入力契約が変わらないため、観測の作り方を変えても判定器は同じままである。

当初は日単位でしか記号を持たないサイトのために `dayLevel` フィールドの導入を検討した。
区分フィルタを正しく操作すれば区分ごとの実値が取れるため、この概念は不要になった。

### 借りるものと借りないもの

`observe.ts` がスクレイパーから借りるのは `prepare` フックだけである。

`prepare` は施設の空き状況テーブルまでページ遷移する手続きであり、記号の意味づけを含まない。
サイトの URL と対象施設への到達経路を知る目的での参照は、既存の設計が明示的に許可している範囲に収まる。

`extract` と `transform` と `STATUS_MAP` は借りない。
これらを借りると、観測が scraper の解釈をなぞることになり、同じ誤りを再現して MATCH を出す。
`symbolMap.ts` が `STATUS_MAP` を import しない理由と同じである。

この線引きは規律だけに委ねず、ユニットテストで検査する（後述）。

### 読み取り戦略

`observe.ts` の内部に、自治体スラッグから読み取り戦略への小さなマップを持つ。
戦略は二つである。

- **direct**：`prepare` 後に描画されている表をそのまま読む。既定の戦略
- **divisionFilter**：「その他の条件で絞り込む」を開き、区分ラベルを選んで「表示」を押す操作を繰り返し、区分ごとに表を読む

切り替える区分ラベルは、`plan.json` の `divisionLabels`（registry の `reservationDivision` の表示ラベル）を使う。
スクレイパー側の `DIVISION_MAP` は借りない。

`divisionFilter` に該当するのは豊島区と江戸川区である。
この二つは同系の予約システムを使っており、フィルタの UI も共通しているため、一つのヘルパーで扱える。

文京区も同系のシステムだが、初回の実行では施設別空き状況の画面で凡例とヘッダだけが描画され、施設名と空き状況のグリッドが現れなかった。
フィルタ操作以前の段階で止まっているため、`divisionFilter` を当てれば読めるかは分からない。
初期実装では `direct` のままとし、`reached: false` が再現するかを見てから戦略を決める。

戦略をスクレイパー側ではなく `spotcheck/` 配下に置くのは、借りている範囲を `prepare` に限るという線引きを保つためである。
スクレイパーに spot check 専用のフックを足すと、フックが増えるたびに境界が曖昧になる。

表構造から戦略を推測する案は採らない。
推測が外れたとき何を読んだのかが分からず、silent failure の検出器が silent failure を起こす。

## 観測の手順

### 対象の絞り込み

`plan.json` の `buildingSystemName` で `scraper.facility(t)` が一致する target を絞る。
さらに `institutionSystemName` を各 target の JSON 表現に対する部分一致で絞る。

一致する target がなければ、建物単位の target とみなして最初の要素を使う。
北区は室ごとに target を持ち、豊島区は建物単位で室を持たないため、両方を一つの手順で扱うにはこのフォールバックが要る。

### 行の特定

表の各行の先頭セルに室名が含まれる行を探す。
見つからなければ `reached: false` とし、読めた行の一覧を note に記録する。

### セルから生記号への変換

セルにテキストがあればそれを、画像があれば `alt` 属性を、`alt` がなければ `src` のファイル名を記号として記録する。

江東区と大田区は空き状況を画像で表示する。
江東区の画像には `alt="予約あり"` のような日本語ラベルが付いており、これは人間が画面を見て読み取る内容と一致する。

### 区分ラベルの扱い

画面に出ているラベルをそのまま記録する。
`observe.ts` では正規化しない。

正規化を観測側で行うと、ラベル対応の誤りが観測の時点で隠れる。
表記ゆれの吸収は `judgeReport.ts` の `normalizeDivisionLabel` が担い、それでも合わない場合はエージェントが `divisionLabels` を見て補正する。

## エラー処理

1 サンプルの失敗が全体を止めないことを原則とする。
spot check は少数サンプルの抽出検査であり、8 件中 1 件が落ちても残りの結論は成立する。

| 状況 | 挙動 |
|---|---|
| target が見つからない | `reached: false`。候補の一覧を note に記録する |
| `prepare` が例外を投げる | `reached: false`。例外メッセージを 500 字まで note に記録する |
| 室の行が見つからない | `reached: false`。読めた行の一覧を note に記録する |
| 対象日の列がない | `dateDisplayed: false`。受付期間と判断できる記載があれば `outOfWindow: true` |
| 区分フィルタの操作が失敗する | その区分だけ欠落させ、note に記録する。他の区分は活かす |

## テスト

実サイトに接続しない範囲で `node --test` によるユニットテストを置く。

1. **盲検の検査**：`observe.ts` のソースに `extract`、`transform`、`STATUS_MAP`、`DIVISION_MAP` への参照がないことを検査する。将来これらを借りた時点で CI が落ちる
2. **セルから記号への変換**：HTML 断片を与え、テキストと `img[alt]` と `img[src]` のそれぞれから期待どおりの生記号が出ることを検査する
3. **戦略マップの整合**：`divisionFilter` に列挙した自治体スラッグが registry に実在することを検査する
4. **出力形状**：生成した観測結果が `ObservedSample` の形を満たすことを検査する

盲検の検査をテストとして置く理由は、この線引きが読み手にとって自然でないことにある。
スクレイパーに `extract` があるのだから使えばよい、という判断は合理的に見える。
その判断が働いた瞬間に spot check は検出器としての価値を失うため、コメントによる注意書きではなく機械的な検査で止める。

## `spot-check.md` の改修

フェーズ 2 の記述を、Playwright MCP でのライブ探索から `observe.ts` の実行に置き換える。

盲検の節に、`observe.ts` が `prepare` だけを借りる設計であることと、その理由を書き足す。

区分フィルタ型サイトについての節を新設する。
豊島区と江戸川区と文京区は、フィルタを操作せずに読むと集約表が読めてしまい、それらしい観測結果が得られる。
初回の実行で実際にこの罠を踏んだため、症状と対処を記録しておく。

エージェントの役割を書き換える。
観測ファイルを全部書く役から、`raw/*.json` を読んで note を補い、異常を記録する役に変える。

江戸川区の総合文化センターが令和 8 年 4 月から令和 9 年 11 月まで全館休館の改修工事中であることは、施設別空き状況の表からは読み取れず、ページ上部のお知らせに書かれていた。
表の記号だけを機械的に集めると、この種の文脈は落ちる。
観測の機械的な部分を決定論スクリプトに移し、文脈の読解をエージェントに残すのは、この違いによる。
